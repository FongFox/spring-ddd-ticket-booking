3 cách tổ chức package
01 Package by Layer project này đang dùng
com.vetautet.ddd
├── controller/
│ ├── resource/
│ │ ├── HiController.java
│ │ └── TicketDetailController.java
│ └── model/ enums/ vo/
├── application/
│ └── service/ticket/
│ ├── ITicketDetailAppService.java
│ ├── cache/TicketDetailCacheService.java
│ └── impl/TicketDetailAppServiceImpl.java
├── domain/ ← business rules, entities
│ ├── model/entity/
│ ├── repository/
│ └── service/
└── infrastructure/ ← DB, Redis, Redisson
    ├── cache/redis/
    ├── distributed/redisson/
    └── persistence/
ưu điểmRanh giới layer rõ ràng. Domain hoàn toàn độc lập. Dễ học DDD.
nhược điểmThêm feature mới phải đụng nhiều package cùng lúc.
dùng khiHọc DDD, project có nhiều layer rõ ràng, cần tách domain khỏi infra.
02 Package by Feature phổ biến ở startup
com.vetautet
├── ticket/ ← toàn bộ ticket feature
│ ├── TicketDetailController.java @RestController
│ ├── ITicketDetailService.java interface
│ ├── TicketDetailServiceImpl.java @Service
│ ├── TicketDetailRepository.java @Repository
│ └── TicketDetail.java @Entity
├── order/
│ ├── OrderController.java
│ ├── IOrderService.java
│ ├── OrderServiceImpl.java
│ ├── OrderRepository.java
│ └── Order.java
└── user/
    ├── UserController.java
    ├── IUserService.java
    └── User.java
ưu điểmFeature mới chỉ đụng 1 package. Dễ tìm theo nghiệp vụ. Team sở hữu feature riêng.
nhược điểmDễ coupling nếu feature gọi nhau. Domain logic có thể bị phân tán.
dùng khiStartup, nhiều team mỗi team sở hữu 1 feature, microservice nhỏ.
03 Package by Model ít phổ biến — tránh dùng cho project lớn
com.vetautet
├── Ticket.java   TicketController.java   TicketService.java   TicketRepository.java
├── Order.java    OrderController.java    OrderService.java    OrderRepository.java
└── User.java     UserController.java     UserService.java     UserRepository.java
ưu điểmĐơn giản. Mọi thứ liên quan đến 1 model nằm cùng chỗ.
nhược điểmProject lớn lên → root package bùng nổ. Khó tìm, khó maintain.
dùng khiProject rất nhỏ, prototype, demo nhanh. Không dùng cho production.
so sánh Khi nào dùng cái nào?
Tiêu chí Package by Layer Package by Feature Package by Model
Project học DDD ✓ Tốt nhất ~ Được ✗ Không
Startup, nhiều feature ~ Được ✓ Tốt nhất ✗ Không
Nhiều team song song ✗ Khó ✓ Tốt nhất ✗ Không
Domain logic rõ ràng ✓ Rõ nhất ~ Phân tán ✗ Lẫn lộn
Prototype / demo nhanh ~ Hơi cồng ~ Được ✓ Nhanh nhất
Scale lên enterprise ~ Cần adjust ✓ Tốt ✗ Không
04 Hybrid — Feature + Layer enterprise, công ty lớn hay dùng
com.vetautet
├── ticket/ ← feature boundary (Package by Feature)
│ ├── controller/ ← layer bên trong feature
│ │ └── TicketDetailController.java @RestController
│ ├── application/
│ │ ├── ITicketDetailAppService.java
│ │ └── TicketDetailAppServiceImpl.java @Service
│ ├── domain/ ← business rules riêng của ticket
│ │ ├── TicketDetail.java @Entity
│ │ ├── ITicketDetailRepository.java
│ │ └── ITicketDetailDomainService.java
│ └── infrastructure/
│ ├── TicketDetailJPAMapper.java
│ └── TicketDetailInfrasRepositoryImpl.java @Repository
├── order/ ← feature boundary
│ ├── controller/
│ ├── application/
│ ├── domain/
│ └── infrastructure/
└── shared/ ← dùng chung giữa các feature
    ├── config/ RedisConfig.java RedissonConfig.java
    └── cache/ RedisInfrasService.java
ưu điểmMỗi team sở hữu 1 feature. Bên trong vẫn rõ layer. Dễ tách thành microservice sau.
nhược điểmSetup phức tạp hơn. Cần shared module cho code dùng chung. Overengineering nếu team nhỏ.
dùng khiNhiều team, project lớn, có kế hoạch tách microservice. Tiki, Shopee, VNG hay dùng pattern này.
Gợi ý: học → Layer | startup → Feature | enterprise → Hybrid
